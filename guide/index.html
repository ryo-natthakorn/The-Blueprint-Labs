The Blueprint Labs

How this was built.

Guide · Nº 01

Twenty-five sites, one method.

The Blueprint Labs is a capability showcase, not a client roster — twenty-five single-scene sites, each built to demonstrate a different code-driven visual technique across a different business vertical. This page documents how the collection was actually assembled: the stack, the repo shape, the batching workflow, and who — or what — did the building.

01 Stack

Every site is plain HTML, CSS, and JavaScript — no framework, no bundler, no build step. The only exception is how the two heavy libraries are loaded:

Three.js

Loaded as a native ES module straight from a CDN via an importmap, which unlocks the full modern addon ecosystem — EffectComposer, post-processing passes, OrbitControls, node materials — rather than the stripped UMD build.

GSAP

Scroll-linked and pointer-driven timelines where a hand-rolled requestAnimationFrame loop would have been more code for the same result.

Raw WebGL

A handful of sites skip Three.js entirely and hand-write GLSL fragment shaders directly against the WebGL context — raymarched fluid, FBM cloud, domain-warped melt.

Canvas 2D

The default for particle systems, generative drawing, and pseudo-3D — cheaper than WebGL for scenes that don't need a real camera.

Web Audio API

A few sites synthesize their own audio live — generative ambient scores, drum machines, turntable pitch-bend — with zero audio files shipped.

Variable fonts

One site is built entirely from font-variation-settings animation — no canvas, no WebGL, just type as the interactive medium.

Fonts are Google Fonts only, icons are Font Awesome free tier, and every image on every site is self-hosted — no live third-party hotlinking, no paid or external AI image/video generation APIs anywhere in the build.

02 Architecture

The whole project is a static monorepo. There's nothing to compile, so deploying to Vercel is zero-config — the repo is the site. 

/sites/01-<slug>/index.html   (+ style.css, script.js, /assets)
… through /sites/25-<slug>/
/guide/index.html               ← you are here
/index.html                     ← the Hub
/projects.json                  ← single source of truth, drives the Hub
/BUILD_LOG.md                   ← one line per completed site
/DESIGN_NOTES.md                ← cross-site palette/motion/type audit

projects.json is the only database this project has — id, title, vertical, style tags, accent color, description, thumbnail, live path, tech used, and which model built it. The Hub reads it client-side and renders every card from it; nothing about a site's metadata lives in HTML markup.

03 Workflow

  1. Phase 1

    Hub shell + schema

    The grid, the filter/sort controls, the projects.json schema, the Playwright screenshot script, and the Vercel config — built before a single site existed.

  2. Phase 2

    The 25 sites

    Built in batches of four to five, autonomously, with three full iteration passes per site — a first pass for the concept, a second for design problems and polish, a third for responsiveness and edge cases. No repeated vertical-plus-style combination across all twenty-five.

  3. Phase 3

    Hub polish + this guide

    Real thumbnails generated against the finished sites, a visual pass on the Hub to sit coherently in front of what it showcases, and this page.

04 Authorship

This is the only page in the project that mentions how the work was made. The twenty-five standalone sites were designed and built by Claude Fable 5, an Anthropic model, working through Phase 2 autonomously in creative batches with a full budget and no cost constraint — the brief was simply to push visual and interactive ambition as far as each concept called for. The Hub shell, this guide, and the final polish pass were done by Claude Sonnet 5. projects.json records which model built which site.
